## ü§ù Contributing

### Adding New Documentation

1. **Choose the right directory**:
   - Architecture decisions ‚Üí `architecture/`
   - Database/Supabase ‚Üí `supabase/`
   - UI/UX improvements ‚Üí `ux/`
   - Feature implementations ‚Üí `features/`
   - Deployment procedures ‚Üí `deployment/`
   - Fixes and troubleshooting ‚Üí `reference/fixes/`
   - Session summaries ‚Üí `summaries/`

2. **Use clear filenames**:
   - Descriptive and specific
   - UPPERCASE with underscores
   - Include context (e.g., `FEATURE_NAME_IMPLEMENTATION.md`)

3. **Include standard sections**:
   - Overview/Summary at top
   - Clear headings
   - Code examples where relevant
   - Links to related documentation

4. **Update indexes**:
   - Add entry to `INDEX.md`
   - Update this README if adding new category

### Documentation Guidelines

- Write for human maintainers, not machines
- Include context and reasoning, not just facts
- Link to related documentation
- Use markdown formatting consistently
- Add dates for time-sensitive information
- Mark deprecated documentation clearly
